# Issue #87 — Двухстоловый режим: спецификация

**Источник:** https://github.com/bon2362/poker-timer/issues/87
**Статус:** Спецификация утверждена, готова к фазе планирования архитектуры

## Контекст

Турниры обычно идут на одном столе (5–8 человек) — это поведение **остаётся по умолчанию**. Иногда играем в большем составе (например, 11 человек) и делимся на 2 стола (5+6). Нужно адаптировать таймер под этот формат, сохранив общий таймер, общие настройки и общий призовой фонд.

## Логика и UX

| # | Решение |
|---|---|
| **Триггер merge** | Авто-пауза + диалог подтверждения при `activePlayers ≤ mergeThreshold`. Появляется **каждый раз** после устранения игрока, пока столы не объединены. На десктопе и в `MobileAdminPanel` |
| **Перемещение игроков** | Manual mid-game через раскрывающееся меню игрока. Без confirm-диалога (всегда можно отменить тем же toggle). Скрыто при `numberOfTables === 1` или после merge |
| **Уведомление о merge** | Минимальное: закрытие диалога + перерисовка GamePanel. Без оверлеев и звуков |
| **Mobile** | `MobileView` — без изменений (списка игроков нет). `MobileAdminPanel` — полная поддержка |
| **Mid-game смена кол-ва столов** | Запрещено. Фиксируется в Setup. Только merge как переход 2→1 |
| **Устранённые в GamePanel** | Общая секция «Вылетели» внизу, без привязки к столу |
| **Undo-вылет** | На актуальный стол. После merge все имеют `tableNumber === 1` |
| **«Минуту!»-таймер** | Глобальный, без изменений (имя + аватар уже идентифицируют игрока) |
| **Один стол опустел до merge** | Разделители остаются — режим формально «2 стола» |
| **Видимость рассадки игрокам** | Не нужна отдельной фичей — таймер и GamePanel видны всем |

## Setup UI (при `numberOfTables === 2`)

- Рядом с каждым выбранным игроком — toggle **«1 / 2»**
- Поле `mergeThreshold` (число активных игроков для предложения объединения)
- Без авто-распределения и кнопок «Случайно» / «Поровну»

## Валидаторы Setup (при `numberOfTables === 2`)

- `mergeThreshold` ∈ `[2, totalPlayers - 1]`
- На каждом столе минимум 2 игрока
- Кнопка «Старт» disabled пока валидаторы не пройдены

## Действие при подтверждении merge

Реализовано через Postgres RPC **`merge_tables(session_id)`** в одной транзакции:

1. Bulk-update `tableNumber = 1` для всех `session_players` (активных и устранённых)
2. `tables_merged_at = now()` на сессии

На клиенте после merge ориентируемся на `sessions.tables_merged_at` (один re-fetch), не на индивидуальные `session_players` events — это снижает дёрганье UI на медленной сети.

## Race condition (два организатора подтвердили одновременно)

- `tables_merged_at` — единая точка истины
- Через Supabase realtime: первое подтверждение → диалог закрывается у всех остальных устройств
- RPC идемпотентен (повторное применение того же значения безопасно)

## Изменения в БД

| Поле | Тип | Дефолт | Назначение |
|---|---|---|---|
| `session_players.table_number` | `INTEGER NOT NULL` | `1` | Номер стола (1 или 2) |
| `sessions.number_of_tables` | `INTEGER NOT NULL` | `1` | Кол-во столов в сессии |
| `sessions.merge_threshold` | `INTEGER NOT NULL` | `0` | Порог автопредложения объединения |
| `sessions.tables_merged_at` | `TIMESTAMPTZ` | `NULL` | Момент фактического объединения |

TS-тип: `tableNumber: number` (не nullable). Существующие сессии можно проигнорировать.

## Изменения в существующих компонентах

- **`TimerDisplay`** — счётчик активных игроков при `numberOfTables === 2 && tables_merged_at === null` показывать как **«4 / 3»** (по столам), иначе как сейчас
- **`GamePanel`** — секции «Стол 1 / Стол 2» с per-section заголовком «Фишек в игре». После merge — единый список
- **`PlayerRow`** (в раскрытом меню) — кнопка «→ Стол 2» / «← Стол 1» при `numberOfTables === 2 && !merged`
- **`SessionSetup`** — toggle «1 / 2» на каждого выбранного игрока + поле `mergeThreshold` + валидаторы
- **`MobileAdminPanel`** — те же изменения что в десктопе (разделители, перемещение, диалог подтверждения)
- **`GameContext`** — методы `movePlayerToTable(sessionPlayerId, table)` и `confirmMerge()`
- **`lib/game.ts`** — `calcGameStats` принимает `tableNumber?: number` для per-table chip count

## Тестирование

- **Unit:** новые методы `GameContext`, валидаторы Setup, обновлённый `calcGameStats`
- **E2E happy-path:** setup 2 столов → старт → устранения → merge dialog → подтверждение → продолжение игры

## План деплоя

1. Миграция БД (ALTER TABLE + создать RPC `merge_tables`)
2. Деплой кода

Порядок строгий: код не должен попасть на прод раньше схемы.

## Не в скоупе MVP (явно отброшено)

- Авто-балансировка / кнопки «Случайно» и «Поровну» в Setup
- Добавление 2-го стола mid-game
- Полноэкранные оверлеи / звуки при merge
- Per-table отображение устранённых
- Двухстоловый режим в `MobileView`
- Запоминание «отказался от merge» (диалог появляется каждый раз)
- Авто-схлопывание разделителей при пустом столе
- Уведомления о перемещении игроков
- Confirm-диалог при перемещении игрока

## Источники анализа

Спецификация получена через 8 уточняющих вопросов и 2 метода advanced elicitation:

- **Cross-Functional War Room** — выявил архитектурный путь и trade-offs
- **What If Scenarios** — поймал edge-case: undo-вылет после merge, race condition, валидация порога
- **Stakeholder Round Table** — добавил ергономику Setup, RPC для атомарности, разбивку счётчика активных в `TimerDisplay`
