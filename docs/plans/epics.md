# Poker Timer — Двухстоловый режим — Epic Breakdown

## Overview

Этот документ декомпозирует требования из [issue-87-spec.md](./issue-87-spec.md) в эпики и сторисы, готовые к реализации. Источник — issue [#87](https://github.com/bon2362/poker-timer/issues/87).

## Requirements Inventory

### Functional Requirements

- **FR1:** Setup позволяет выбрать количество столов (1 или 2). По умолчанию — 1
- **FR2:** При выборе 2 столов Setup показывает toggle «1 / 2» рядом с каждым выбранным игроком для назначения стола
- **FR3:** При выборе 2 столов Setup показывает поле ввода `mergeThreshold` с валидацией `[2, totalPlayers - 1]`
- **FR4:** Setup-валидаторы блокируют кнопку «Старт»: минимум 2 игрока на каждом столе + корректный `mergeThreshold`
- **FR5:** Таймер блайндов и перерывов общий для обоих столов (существующая логика без изменений)
- **FR6:** Все денежные настройки (buy-in, rebuy, addon, prizes) общие для обоих столов
- **FR7:** GamePanel показывает разделители-секции «Стол 1 / Стол 2» с per-section счётчиком фишек при `numberOfTables === 2 && tables_merged_at === null`
- **FR8:** GamePanel показывает общий банк и призовые независимо от количества столов
- **FR9:** Устранённые игроки отображаются в общей секции «Вылетели» внизу GamePanel, без привязки к столу
- **FR10:** TimerDisplay показывает счётчик активных игроков как «N1 / N2» (по столам) при `numberOfTables === 2 && tables_merged_at === null`, иначе общим числом
- **FR11:** При устранении игрока, если `activePlayers ≤ mergeThreshold` и столы ещё не объединены, таймер автопаузится и показывается диалог «Объединить столы?»
- **FR12:** Диалог объединения отображается на десктопе и в `MobileAdminPanel`
- **FR13:** При подтверждении merge выполняется RPC `merge_tables(session_id)`: bulk-update `table_number = 1` для всех `session_players` + `tables_merged_at = now()` на сессии
- **FR14:** При отказе от merge диалог закрывается, таймер возобновляется. При следующем устранении (если порог по-прежнему достигнут) диалог появляется снова
- **FR15:** После merge GamePanel рендерится без разделителей; кнопки перемещения игроков скрываются
- **FR16:** В раскрывающемся меню активного игрока доступна кнопка «→ Стол 2» / «← Стол 1» при `numberOfTables === 2 && !merged`
- **FR17:** Перемещение игрока через кнопку обновляет `tableNumber` без подтверждения (отмена — обратным toggle)
- **FR18:** Undo-вылет возвращает игрока с актуальным `tableNumber` (после merge — всегда 1)
- **FR19:** «Минуту!»-таймер остаётся глобальным независимо от количества столов
- **FR20:** `MobileView` не изменяется (нет списка игроков)
- **FR21:** Mid-game смена количества столов запрещена (только merge как переход 2 → 1)

### NonFunctional Requirements

- **NFR1:** Single-table сценарий (`numberOfTables === 1`) работает без регрессии — поведение по умолчанию
- **NFR2:** Merge выполняется атомарно через Postgres RPC (либо всё, либо ничего)
- **NFR3:** Race condition при одновременном подтверждении merge разрешается через `tables_merged_at` как единую точку истины (RPC идемпотентен)
- **NFR4:** После merge клиент ориентируется на `sessions.tables_merged_at` (один re-fetch), не реагируя на индивидуальные `session_players.UPDATE` events
- **NFR5:** Перезагрузка страницы во время диалога merge не теряет состояние — диалог пересчитывается из условий (`numberOfTables > 1 && activePlayers ≤ mergeThreshold && !merged`)
- **NFR6:** Существующие сессии в проде можно проигнорировать (миграция с `DEFAULT 1` безопасна)

### Additional Requirements

- **БД миграция (4 ALTER TABLE):**
  - `session_players.table_number INTEGER NOT NULL DEFAULT 1`
  - `sessions.number_of_tables INTEGER NOT NULL DEFAULT 1`
  - `sessions.merge_threshold INTEGER NOT NULL DEFAULT 0`
  - `sessions.tables_merged_at TIMESTAMPTZ NULL`
- **Postgres RPC** `merge_tables(p_session_id UUID)` — атомарная транзакция
- **TS-типы:** `tableNumber: number` (не nullable, default 1) на `SessionPlayer`; `numberOfTables: number`, `mergeThreshold: number`, `tablesMergedAt: string | null` на `Session`
- **`lib/game.ts`:** `calcGameStats` принимает опциональный `tableNumber` для per-table chip count
- **План деплоя:** строгий порядок миграция → деплой кода
- **E2E:** добавить Playwright happy-path для двухстолового сценария

### UX Design Requirements

- **UX-DR1:** Toggle «1 / 2» рядом с именем игрока в `SessionSetup`, появляется только при `numberOfTables === 2`
- **UX-DR2:** Поле ввода `mergeThreshold` (number input) в `SessionSetup`, видимое только при `numberOfTables === 2`, с подсказкой допустимого диапазона
- **UX-DR3:** Sticky-заголовки секций «Стол 1» / «Стол 2» в `GamePanel`, стилистика как существующие headers (uppercase, `#555`, tracking `[2px]`)
- **UX-DR4:** Per-section мини-stat «Фишек в игре N» внутри каждой секции стола (визуально аналогично существующему общему счётчику)
- **UX-DR5:** Кнопка `→ Стол 2` / `← Стол 1` в раскрытом меню `PlayerRow`, стилистически согласованная с существующими кнопками меню («Вылетел:а», «Минуту!»)
- **UX-DR6:** Модальный диалог «Объединить столы?» с двумя кнопками («Объединить» — primary, «Отмена» — secondary). Поверх таймера. Реализован одинаково на десктопе и в `MobileAdminPanel`
- **UX-DR7:** В `TimerDisplay` счётчик активных при `numberOfTables === 2 && !merged` показывает «N1 / N2» с тонким разделителем

### FR Coverage Map

- FR1, FR2, FR3, FR4, FR21 → Epic 1 (запуск сессии на двух столах)
- FR5, FR6 → Epic 1 (по факту no-op, унаследовано)
- FR7, FR8, FR9, FR10, FR16, FR17, FR19, FR20 → Epic 2 (игровой экран)
- FR11, FR12, FR13, FR14, FR15, FR18 → Epic 3 (объединение столов)
- NFR1, NFR6 → проверяется в каждом эпике через регрессию
- NFR2, NFR3, NFR4, NFR5 → Epic 3
- UX-DR1, UX-DR2 → Epic 1 / Story 1.2
- UX-DR3, UX-DR4, UX-DR7 → Epic 2 / Stories 2.1, 2.2
- UX-DR5 → Epic 2 / Story 2.3
- UX-DR6 → Epic 3 / Stories 3.1, 3.3

## Epic List

### Epic 1: Запуск сессии на двух столах
Организатор может настроить и стартовать сессию с двумя столами: указать количество, рассадить игроков и задать порог объединения.
**FRs covered:** FR1, FR2, FR3, FR4, FR5, FR6, FR21

### Epic 2: Игровой экран с двумя столами
Игроки и организатор видят корректную раздельную информацию по столам во время игры — фишки, активных игроков, и могут перемещать игроков между столами.
**FRs covered:** FR7, FR8, FR9, FR10, FR16, FR17, FR19, FR20

### Epic 3: Объединение столов
Когда количество активных игроков достигает порога, игра автоматически останавливается, организатор подтверждает объединение, после чего сессия продолжается как одностоловая.
**FRs covered:** FR11, FR12, FR13, FR14, FR15, FR18

## Epic 1: Запуск сессии на двух столах

**Goal:** Организатор открывает SessionSetup, выбирает «2 стола», распределяет выбранных игроков по столам через toggle «1/2», задаёт порог объединения и стартует. БД и типы поддерживают новые поля.

### Story 1.1: Расширить БД схему и TypeScript-типы

As a разработчик,
I want добавить поля для двухстолового режима в БД и типы,
So that последующие сторисы могли опираться на стабильную схему.

**Acceptance Criteria:**

**Given** существующая БД Supabase с таблицами `sessions` и `session_players`
**When** применяется новая миграция
**Then** на `session_players` появляется колонка `table_number INTEGER NOT NULL DEFAULT 1`
**And** на `sessions` появляются колонки `number_of_tables INTEGER NOT NULL DEFAULT 1`, `merge_threshold INTEGER NOT NULL DEFAULT 0`, `tables_merged_at TIMESTAMPTZ NULL`
**And** Postgres RPC-функция `merge_tables(p_session_id UUID)` создана и идемпотентна

**Given** обновлённая схема БД
**When** клиентский код собирается
**Then** в `types/game.ts` `SessionPlayer` имеет поле `tableNumber: number` (default 1)
**And** `Session` имеет поля `numberOfTables: number`, `mergeThreshold: number`, `tablesMergedAt: string | null`
**And** `lib/supabase/sessions.ts` и `lib/supabase/players.ts` корректно сериализуют/десериализуют новые поля

**Given** single-table сессия запущена до миграции
**When** клиент загружает её после миграции
**Then** все session_players имеют `tableNumber === 1`
**And** сессия имеет `numberOfTables === 1`, `tables_merged_at === null`
**And** существующая single-table логика работает без регрессии (NFR1)

### Story 1.2: SessionSetup UI для выбора количества столов и рассадки

As a организатор турнира,
I want в SessionSetup выбрать количество столов и распределить игроков,
So that я мог стартовать сессию в формате 1 или 2 столов.

**Acceptance Criteria:**

**Given** открыт SessionSetup
**When** я вижу настройки сессии
**Then** доступен переключатель «Кол-во столов» со значениями 1 / 2 (по умолчанию 1)

**Given** я выбрал «2 стола»
**When** я смотрю на список выбранных игроков
**Then** рядом с каждым игроком отображается toggle «Стол 1 / Стол 2» (UX-DR1)
**And** появляется поле ввода «Порог объединения» (UX-DR2) с подсказкой допустимого диапазона `[2, totalPlayers - 1]`

**Given** я выбрал «1 стол»
**When** я смотрю на список игроков
**Then** toggle и поле порога не отображаются
**And** при старте сессии все session_players имеют `tableNumber === 1`, `numberOfTables === 1`

**Given** я переключился с «2 стола» на «1 стол»
**When** старт сессии
**Then** значения toggle игроков игнорируются (всем `tableNumber === 1`)

### Story 1.3: SessionSetup валидаторы двухстолового режима

As a организатор турнира,
I want получать понятные ошибки при невалидной конфигурации двухстолового режима,
So that я не стартовал сессию в неработающем состоянии.

**Acceptance Criteria:**

**Given** выбрано «2 стола»
**When** на одном из столов меньше 2 игроков
**Then** кнопка «Старт» disabled
**And** под полем рассадки видно поясняющее сообщение об ошибке

**Given** выбрано «2 стола»
**When** `mergeThreshold` пустой, или `< 2`, или `≥ totalPlayers`
**Then** кнопка «Старт» disabled
**And** под полем порога видна ошибка с допустимым диапазоном

**Given** все валидаторы пройдены
**When** я нажимаю «Старт»
**Then** сессия создаётся с корректными `numberOfTables`, `mergeThreshold` и `tableNumber` для каждого session_player

## Epic 2: Игровой экран с двумя столами

**Goal:** Во время сессии с двумя столами `GamePanel` отображает игроков с разделителями по столам, `TimerDisplay` показывает раздельный счётчик активных, организатор может перемещать игроков между столами через раскрывающееся меню.

### Story 2.1: GamePanel — разделители «Стол 1 / Стол 2» и per-section фишки

As a организатор турнира,
I want видеть в GamePanel игроков сгруппированными по столам,
So that я понимал состав каждого стола в любой момент.

**Acceptance Criteria:**

**Given** активная сессия с `numberOfTables === 2 && tables_merged_at === null`
**When** GamePanel открыт
**Then** активные игроки сгруппированы под заголовками «Стол 1» и «Стол 2» (UX-DR3)
**And** каждая секция имеет собственный счётчик «Фишек в игре N» (UX-DR4) — `calcGameStats` с фильтром по `tableNumber`
**And** общий банк и призовые отображаются в шапке без изменений (FR8)
**And** устранённые игроки в общей секции «Вылетели» внизу — без привязки к столу (FR9)

**Given** активная сессия с `numberOfTables === 1` или с заполненным `tables_merged_at`
**When** GamePanel открыт
**Then** игроки в едином списке, без разделителей (NFR1, FR15)
**And** счётчик «Фишек в игре» — общий, как сейчас

**Given** активная сессия с двумя столами, на одном из них 0 активных
**When** GamePanel открыт
**Then** разделители остаются видимыми (пустая секция показывается с заголовком и нулевым счётчиком)

### Story 2.2: TimerDisplay split-счётчик активных игроков

As a организатор турнира,
I want видеть в TimerDisplay раздельный счётчик активных по столам,
So that я понимал баланс столов не открывая GamePanel.

**Acceptance Criteria:**

**Given** активная сессия с `numberOfTables === 2 && !merged`
**When** TimerDisplay рендерится
**Then** счётчик активных показывается как «N1 / N2» (UX-DR7)

**Given** активная сессия с `numberOfTables === 1` или после merge
**When** TimerDisplay рендерится
**Then** счётчик активных показывается общим числом, как сейчас (NFR1)

### Story 2.3: Перемещение игрока между столами через меню PlayerRow

As a организатор турнира,
I want перемещать активного игрока между столами в один клик,
So that я мог балансировать столы при необходимости.

**Acceptance Criteria:**

**Given** активная сессия с `numberOfTables === 2 && !merged`, активный игрок Иван на столе 1
**When** я раскрываю меню Ивана (клик по имени) и нажимаю «→ Стол 2»
**Then** `tableNumber` Ивана становится 2 без диалога подтверждения (FR17, UX-DR5)
**And** GamePanel перерисовывается: Иван теперь под заголовком «Стол 2»

**Given** активный игрок Иван на столе 2
**When** я раскрываю меню Ивана и вижу кнопку «← Стол 1»
**Then** кнопка стилистически согласована с существующими «Вылетел:а» и «Минуту!» (UX-DR5)

**Given** сессия с `numberOfTables === 1` или после merge
**When** я раскрываю меню любого игрока
**Then** кнопки перемещения отсутствуют (FR15, FR16)

**Given** «Минуту!»-таймер запущен на любом игроке
**When** я смотрю на оверлей
**Then** оверлей глобальный, без изменений (FR19)

## Epic 3: Объединение столов

**Goal:** При достижении порога организатор автоматически получает диалог «Объединить столы?», подтверждение запускает атомарный merge через Postgres RPC. После merge UI и БД-состояние согласованы; race conditions и перезагрузки безопасны.

### Story 3.1: Auto-pause + диалог merge на десктопе

As a организатор турнира,
I want получать предложение объединить столы при достижении порога,
So that я не следил за счётчиком вручную.

**Acceptance Criteria:**

**Given** активная сессия с `numberOfTables === 2 && !merged`, `mergeThreshold === 8`
**When** игрок устранён и `activePlayers === 8`
**Then** таймер автоматически ставится на паузу (FR11)
**And** появляется модальный диалог «Объединить столы?» с кнопками «Объединить» и «Отмена» (UX-DR6)

**Given** диалог открыт
**When** я нажимаю «Отмена»
**Then** диалог закрывается
**And** таймер возобновляется
**And** `tables_merged_at` остаётся `null`

**Given** я отменил диалог, активных всё ещё ≤ порога
**When** устранён следующий игрок
**Then** диалог появляется снова с авто-паузой (FR14)

**Given** страница перезагружена во время открытого диалога
**When** клиент загрузил состояние
**Then** диалог появляется снова, потому что условия `activePlayers ≤ mergeThreshold && !merged` всё ещё выполняются (NFR5)

### Story 3.2: Реализация merge через RPC и обработка race condition

As a разработчик,
I want чтобы подтверждение merge выполнялось атомарно через Postgres RPC,
So that я не получил частичных состояний и race conditions между устройствами.

**Acceptance Criteria:**

**Given** диалог merge открыт на двух устройствах
**When** организатор нажимает «Объединить» на любом из них
**Then** клиент вызывает RPC `merge_tables(session_id)`
**And** в одной транзакции: все `session_players.table_number` становятся 1, `sessions.tables_merged_at = now()` (FR13, NFR2)

**Given** одновременное подтверждение с двух устройств
**When** оба клиента вызывают RPC
**Then** результат идемпотентный (повторное применение того же значения безопасно) (NFR3)
**And** оба устройства получают финальное согласованное состояние

**Given** RPC завершилась успешно
**When** Supabase realtime отдаёт `sessions.UPDATE` с непустым `tables_merged_at`
**Then** клиент делает один re-fetch активной сессии (NFR4)
**And** не пытается реагировать на каждое отдельное `session_players.UPDATE` событие из bulk-апдейта

### Story 3.3: Диалог merge в MobileAdminPanel

As a организатор турнира на телефоне,
I want видеть и подтверждать диалог объединения в админ-панели,
So that мне не нужно бежать к десктопу для подтверждения.

**Acceptance Criteria:**

**Given** открыт `MobileAdminPanel` на сессии с двумя столами
**When** условия для merge выполнены
**Then** появляется тот же диалог «Объединить столы?», что и на десктопе (FR12, UX-DR6)
**And** диалог поверх админ-панели, не блокирует видимость основного состояния

**Given** диалог открыт на десктопе и в `MobileAdminPanel` одновременно
**When** на одном из них нажата «Объединить»
**Then** на другом устройстве диалог закрывается автоматически (через realtime событие об обновлении `tables_merged_at`)

### Story 3.4: Поведение после merge

As a организатор турнира,
I want чтобы после объединения интерфейс корректно отражал одностоловое состояние,
So that не было лишних разделителей, кнопок и путаницы с прежней рассадкой.

**Acceptance Criteria:**

**Given** сессия только что объединена (`tables_merged_at` заполнен)
**When** GamePanel рендерится
**Then** разделители «Стол 1 / Стол 2» исчезают, единый список (FR15)
**And** счётчик «Фишек в игре» — общий

**Given** объединённая сессия
**When** я открываю раскрывающееся меню активного игрока
**Then** кнопки «→ Стол 2» / «← Стол 1» отсутствуют (FR16)

**Given** объединённая сессия, ранее устранённый Иван имел `tableNumber === 2` до merge
**When** я нажимаю «↩ вернуть» на Иване
**Then** Иван возвращается со статусом `playing`
**And** его `tableNumber === 1` (потому что merge сделал bulk-update для всех session_players, включая устранённых) (FR18)

**Given** объединённая сессия
**When** TimerDisplay рендерится
**Then** счётчик активных — общим числом (FR10, FR15)

### Story 3.5: E2E happy-path тест двухстолового сценария

As a разработчик,
I want иметь Playwright-тест, проверяющий полный двухстоловый цикл,
So that я мог рефакторить код без страха регрессии в этой фиче.

**Acceptance Criteria:**

**Given** запущен `npx playwright test`
**When** выполняется новый сценарий `e2e/desktop/two-tables.spec.ts`
**Then** тест проходит шаги: setup 6 игроков, выбор «2 стола», рассадка 3+3, threshold=4
**And** старт сессии, последовательное устранение игроков
**And** при достижении threshold появляется диалог merge
**And** подтверждение merge, проверка единого списка в GamePanel
**And** продолжение игры до победителя

**Given** тестовый прогон
**When** проверяется регрессия single-table сценария
**Then** существующие e2e тесты не сломаны (NFR1)
